# Index Singletons With Name-Literal Types

Interaction indices are module-level singleton values: each shipped index is the only instance of a hidden class parameterized by its name literal â€” ``class _SII(Index[Literal["SII"]])`` with ``SII: Final = _SII()`` â€” and the exported value is at once what users pass, what dispatch compares against with ``is``, and what explanations record. This replaces ADR 0011's classes-passed-uninstantiated representation, which forced the ``type[...]`` spelling into every annotation (``type[SV | FSII | FBII]``, ``TypeIs[type[T]]``) and pushed class-object protocol conformance to the edge of what type checkers model. The user grammar is unchanged (``ExactExplainer(game, SII, order=2)``); everything ADR 0011 decided about explainer-owned orders, ``resolve_order``, exact-singleton dispatch pinning, and the capability declarations stands.

The name literal in the type is the single source of the name: ``Index.__init_subclass__`` reads it from ``__orig_bases__``, so ``Index[Literal["k-SII"]]`` *is* the declaration that the index is called k-SII, and the ``name`` property returning the literal type places the parameter in an output position, making it covariant. Covariance is what turns closed support sets into ordinary type expressions â€” ``Regression`` accepts ``Index[Literal["SV", "FSII", "FBII"]]`` (exported as ``KernelRegressionIndex``), ``PermutationSampling`` accepts ``PermutationIndex``, and the exact explainer's ``ExactIndex`` unions the capability protocols with ``Index[Literal["k-SII"]]`` for the one dedicated solver without a capability â€” so passing ``SII`` where only kernel-matched indices work is a static error again, with no second name anywhere: annotations use ``Index[...]`` aliases, checks use the singleton (``index is FBII``, ``isinstance(index, type(KADDSHAP))``), and the hidden classes are never spoken. The hidden class's constructor returns the singleton (per-class ``__new__``), so manual construction, ``copy``, ``deepcopy``, and pickling all preserve the identity that dispatch relies on; calling the singleton itself (``SII(order=2)``, the old-grammar reflex) raises a teaching error whose example adapts to the index's order rule. Capability protocols are untouched and now describe plain instances â€” the one representation every type checker models natively â€” and third-party indices remain structural: any conforming value works with the capability-dispatched explainers without inheriting ``Index``.

Two alternatives were probed and rejected with evidence. Enum members are the only stdlib values admissible in type expressions (``Literal[SV, FSII]`` even works with re-exported bare names, verified under ty), but one enum class gives every member every method, so capability conformance collapses â€” ``isinstance(SII, RegressionIndex)`` would hold for all members â€” and the member set is closed against third-party indices; the taxonomy is worth more than the prettier spelling. Self-instance classes (an object that is an instance of itself, ``type``-style) die at runtime: CPython restricts ``__class__`` assignment on ``type`` subclasses, and ty rejects the construction regardless. A third option, nameable sentinel classes (``ShapleyValue`` + ``SV = ShapleyValue()``), was rejected by the maintainers before probing: it preserves static precision but forces a second name into annotations and checks, which is the ergonomic regression this decision exists to avoid.
