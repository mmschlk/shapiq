# Open Names, Extensional Equality, And Value Preservation

*Superseded in part by [ADR 0011](0011-index-classes-and-explainer-owned-orders.md): extensional equality (the second paragraph) was removed together with per-instance orders. Open names and ``preserves_value`` stand.*

Interaction index names are open strings. The protocol types ``name`` as ``str``, the metadata validator accepts any non-empty name, and the per-name sanity rules apply only to shipped names; ``InteractionIndexName`` remains as the Literal vocabulary of shipped names for documentation and the shipped classes' own annotations. This supersedes the closed-set deferral recorded in ADRs 0005 and 0008 and makes their extension claim true: a third-party index providing a capability now works end to end with the exact explainer under its own name, with that name carried faithfully on explanations. Name strings appear in even fewer places as a result — behavior selection was already type- and capability-based, and now metadata validation no longer holds a name registry either.

Index objects compare extensionally, superseding ADR 0008's rejection of cross-type equality with a refined semantics that resolves its objections. Equality is equality of attribution rules on nonempty interactions: an instance constructed at order one collapses onto the probabilistic value it generalizes, so ``SII(order=1) == SV() == CHII(order=1) == FSII(order=1)`` — one equivalence class per generalized value, which is exactly what the mathematics says, the chaining case included. All other instances compare by type and parameters, so ``SII(order=2) != CHII(order=2)`` and ``SII(order=2) != SII(order=1)``: the order-1 *block of* a higher-order explanation is not the value for identity-semantics indices, and the equality deliberately does not claim it is. Hashes follow the same canonical identity, so sets and dicts collapse the equivalence class consistently; estimator dispatch is unaffected because it keys on index types, and the closed entry-point unions still describe implementation strategies per type — ``Regression`` accepts ``SV()`` but not the extensionally equal ``KADDSHAP(order=1)``, because support is about having a sampler and solver for the type, not about the measurement. Two caveats are part of the contract: order-0 conventions stay per-index (``SII(order=1)`` carries none; FBII's is a fitted intercept that differs numerically from ``BV()``'s empty value), and game-dependent identities such as k-SII at full order coinciding with the Moebius transform remain out of scope, as before. The transitivity, substitutability, and hashing objections of ADR 0008 were raised against an unscoped equality; scoping it to order-one instances and nonempty interactions, with canonical hashing, is the version the design owner chose to model. The equality lives in a shared ``ExtensionalEquality`` base — implementation sharing, exported for third-party index authors, not the between-index inheritance rejected in ADR 0008, which remains rejected: the generalization relation is many-to-one and stays declared data.

Order semantics gains an orthogonal axis instead of a third value. The binary coverage/identity flag describes whether attributions of shared interactions change with the order, and kADD-SHAP showed it is too coarse alone: its pair attributions change with the order (identity) while its order-1 attributions provably remain Shapley values at every order. ``preserves_value`` declares exactly that — whether the generalized value stays readable off the order-1 attributions at any order. All coverage indices preserve trivially; kADD-SHAP is the interesting case, identity yet preserving; STII, k-SII, FSII, FBII, and JointSV do not preserve. A three-valued semantics enum was rejected because the third value would describe only the order-1 block and muddy what the axis measures; a per-size stability method was rejected as machinery without a consumer. Both directions of every declaration are tested numerically: preserving indices must match their value at order two, non-preserving ones must not.
