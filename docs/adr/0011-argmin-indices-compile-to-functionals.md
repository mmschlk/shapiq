# Argmin Indices Compile To Coalition Functionals

The regression indices were never a second primitive. A weighted least squares solution operator is linear in the game, and every fit shapiq ships is invariant under player permutations, so its coefficient on ``v(K)`` can only depend on the cardinalities ``|K & S|`` and ``|K|`` — the orbits of interaction–coalition pairs under the symmetric group. That is exactly the factored form of a coalition functional. This decision makes the consequence structural: FSII, FBII, and kADD-SHAP declare an argmin specification (row kernel per coalition size, cardinality basis whose design value depends only on interaction and intersection size, exact-interpolation flags for the endpoints), and ``derive_functional`` compiles the specification into the same coefficient tables that cardinal weights and bloc marginals produce directly. The cardinal, generalized-value, argmin, and aggregation capabilities are now presentations of one canonical object, and the calculus has a single semantic layer: everything is a table.

Compilation needs no closed forms and no association-scheme analysis. The solution operator's columns are its responses to indicator probe games, one probe per coalition size by symmetry, and each probe problem is invariant under the stabilizer of its probe coalition — so the fit collapses onto cardinality classes: rows become (inside, outside) coalition profiles weighted by kernel times multiplicity, unknowns become (size, intersection) interaction classes, constraints collapse with them, and each probe is a small equality-constrained weighted least squares solve in float64, ``n + 1`` solves of dimension a few dozen in total. The compiled KernelSHAP table equaling the Shapley value's cardinal table coefficient for coefficient is now an executed test rather than a cited theorem, and the previously separate exact solvers for FSII, FBII, and kADD-SHAP were deleted outright: the exact explainer derives a functional and contracts it, for every index, with the existing exact suite passing unchanged as the regression check. Uniqueness is the one assumption — the invariance argument needs the fit's solution to be unique, so orders extreme enough to make the design rank deficient are outside the compiled contract (they were min-norm artifacts under the old solvers too).

The practical payoff lands on FBII. Its unconstrained fit with a free intercept never matched the constrained-interpolation regression contract, so it had dedicated exact code and no sampled estimator at all; as a compiled functional the intercept is simply the size-0 column and the uniform kernel spans the endpoints, so FBII gains an unbiased anytime ``MonteCarlo`` estimator without any FBII-specific code. FSII and kADD-SHAP gain the same estimator as an alternative to the kernel-matched ``Regression``, and the two now sit in a clean trade-off: the compiled-functional Monte Carlo estimate is unbiased at every budget, while the sampled regression is consistent, exact for ``k``-additive games from identification onward, and typically lower variance — both derived from the same declaration. ``RegressionIndex`` therefore survives as the narrower contract of the kernel-matched estimator (constrained interpolation, endpoint-zero kernel), while ``ArgminIndex`` is the compilation contract; ``define_regression_index`` implements both, so a defined kernel gets both estimators.

Costs accepted: compilation is polynomial but loops in Python (irrelevant next to game evaluations at prototype scale; vectorization is routine future work), collapsed multiplicities overflow float64 somewhere beyond several hundred players, and the whole construction assumes permutation equivariance — a directed or player-weighted index will need either its own presentation or an equivariance-breaking extension of the table's index set.
