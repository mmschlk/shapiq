# Index Classes And Explainer-Owned Orders

*Superseded in part by [ADR 0012](0012-index-singletons-with-name-literal-types.md): the representation moved from classes-passed-uninstantiated to module-level singleton values with name-literal types, removing the ``type[...]`` spelling. Explainer-owned orders, ``resolve_order``, dispatch pinning, and the capability additions stand.*

Interaction indices are plain classes used as namespaces of classmethods and passed to explainers as the class itself — ``ExactExplainer(game, SII, order=2)`` — and the explanation order is owned by the explainer, not the index. An index is an attribution rule; game theory does not parameterize the rule by a truncation order, so the order is a property of the requested explanation. Each index still owns the *knowledge* about orders through a ``resolve_order`` classmethod: probabilistic values (SV, BV) fix the order at one, the transforms (Moebius, Co-Moebius) default to all orders, and every other index requires an explicit order — there is no silent default order anymore. This is also the extension point for indices with genuinely restricted order support (an index defined only for one top order can reject others), which the previous order-as-field design could not express. Weight capabilities take the resolved order as an argument (``derivative_weights(n_players, size, order=k)``), which makes the coverage/identity distinction visible in code: coverage-semantics indices ignore the order argument, identity-semantics indices depend on it. Instantiating an index raises a teaching error whose example adapts to the index's order rule; the enforcement lives in a metaclass that overrides ``type.__call__``, so the instantiation machinery itself — ``__new__`` and ``__init__`` included — is unreachable and ``repr`` prints the bare name: the class is the value, Python's closest spelling of a Scala ``object``. (The purer spelling — a class-statement hook inheriting from ``object`` that returns a non-class singleton — works at runtime but was rejected: ty keeps treating the name as a class while the runtime value is an instance, ``issubclass`` and ``type[...]`` annotations stop existing, and the static story the entry points rely on collapses.) Entry points state their supported indices as ``type[...]`` unions over the capability protocols, so index support stays statically visible and third-party capability classes type-check at call sites.

This supersedes three earlier decisions in part. ADR 0005's "order lives uniformly on the index" and its immutable-value-object representation are replaced: the single-source-of-truth argument that motivated order-on-the-index is preserved (the explainer's resolved ``order`` is now the one source), while the index keeps only order *constraints*. ADR 0007's rider moving interaction orientation onto index objects is reverted, and orientation is deleted from indices and explainers entirely: whether interaction keys are sets or ordered tuples is a property of how an explanation represents its keys, not a game-theoretic property of an index, so ``orientation`` survives only as a representation field on explanation arrays (every shipped explainer produces undirected explanations). ADR 0009's extensional equality is removed wholesale: without per-instance orders there are no order-1 instances to collapse, classes compare by identity, and dispatch tables key directly on the index classes. ADR 0008's original rejection of cross-index equality is thereby restored, in the stronger form of the question no longer being expressible; the order-1 relationship lives on as the ``generalizes`` declaration (now referencing the value class), which remains tested numerically, together with ``preserves_value`` and the open name set from ADR 0009, all of which stand.

Two capability declarations were added while restructuring. FBII declares its regression kernel — the uniform Banzhaf kernel with unit weight at every coalition size — so its membership in the regression family is structural rather than special-cased; nonzero end weights are the declared signal that a kernel fits every coalition with a free intercept instead of interpolating the endpoints as constraints, and the exact constrained solver rejects such kernels with a teaching error. SV and BV declare singleton bloc-marginal weights, making explicit that each probabilistic value is the size-one restriction of its generalized value (SGV, BGV). Dispatch is pinned per family: the closed entry points (Regression, PermutationSampling) and the dedicated exact solvers key on the exact class and reject proper subclasses of shipped indices with teaching errors — a subclass overriding a kernel must never be silently routed to a solver or sampler that ignores the override — while the capability paths of the exact explainer stay structural and honor overrides. The cardinality basis of the weight capabilities was reviewed and kept: weights depending only on the sizes of the interaction and the outside coalition are the symmetry shared by every shipped index, and player-position-specific weightings (weighted Shapley values, Owen values under coalition structure) are a future, separate capability rather than a generalization of the cardinal one.
