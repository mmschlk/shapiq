# Interaction Index Objects

*Superseded in part by [ADR 0011](0011-index-classes-and-explainer-owned-orders.md): indices are now plain classes passed uninstantiated, and the explanation order moved to the explainer. Type-and-capability dispatch and names-as-metadata stand.*

Interaction indices are immutable value objects — `SV()`, `BV()`, `SII(order=2)`, `BII(order=2)`, `STII(order=2)`, `FSII(order=2)` — passed to explainers in place of string names. Explainers select behavior by index type and capability, never by name: `WeightedDerivativeIndex` marks indices whose attributions are weighted sums of discrete derivatives and supplies the cardinal weights, and `RegressionIndex` marks indices defined by a kernel-weighted least squares fit and supplies the kernel, with the empty and grand coalition interpolated exactly as constraints. The exact explainer accepts the union of both capabilities, so a third-party index providing either method works without shapiq changes, and the type checker answers which explainer supports which index at construction time. String names remain explanation metadata; index-specific validation lives on the objects. The central metadata validator keeps its per-index rules only as sanity checks for hand-built explanation arrays, which still carry plain names and orders — no explainer path can violate them anymore.

Order lives uniformly on the index, defaulting to two wherever it exists; SV and BV carry no order field, so requesting them at another order is unrepresentable rather than validated. The alternative — keeping order off SII and BII because their attributions are order-independent — was rejected: users naturally request "SII up to order two", a single location for order removes the possibility of conflicting values in two places, and an absent parameter teaches nothing. The real distinction is carried as data instead: every index declares its order semantics as coverage (attributions of shared interactions are unchanged across orders — SV, BV, SII, BII) or identity (attributions change with the order — STII, FSII), so documentation teaches the interpretation and later tooling can warn when explanations of identity indices are compared across orders. `Explainer.order` derives from the index, leaving one source of truth.

Estimator families whose implementations are genuinely index-specific keep one class per index for now: the permutation approximators construct their index objects internally, and their sampled streams are unchanged. The intended end state is one entry point per method family with the index object dispatching to the concrete implementation, which this decision prepares by keying all internal dispatch on types and capabilities. k-SII left the accepted name set until it exists: it is an explanation-level transform of SII, not an estimator target, and a name the type system advertises should be a name some explainer accepts. Explanation metadata keeps a closed name Literal; open third-party naming is a known limit deferred until a real external index forces the question.

Alongside the migration, the exact explainer moved from host-side numpy float64 to pure JAX in the game's value precision: float32 under JAX defaults, float64 when the application enables x64. This is safe for the faithful indices because both FSII solves — exact and sampled — now substitute the empty- and grand-coalition equality constraints out of the least squares system instead of approximating them with big-M rows. Elimination is exact where big-M was approximate, removes the constraint-weight scaling that kept the sampled estimator consistent, and keeps the reduced system as well conditioned as the design itself, so float32 solves are safe; identification is equivalent to the previous rank condition, leaving minimum budgets unchanged.
